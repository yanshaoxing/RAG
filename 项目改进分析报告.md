# RAG 项目改进分析报告

> 分析日期：2026-07-22
> 分析范围：全部源码（`app/`、`rag/`、`scripts/`、`indexer.py`、`rag/config.py`）及 `data/` 产物结构
> 结论先行：**架构设计合理，不需要推倒重写；需要的是"缺陷修复 + 有针对性的局部重构 + 工程化补课"。**

---

## 一、总体评估

### 1.1 值得肯定的设计

这个项目的整体架构思路在同类内部工具中是相当出色的：

- **分阶段索引管线**（chunks → 摘要树 → BM25 → 向量 → 图谱，各阶段独立持久化）——失败后删目录续跑的思路正确；
- **三路查询改写**（NL 改写 / HyDE / 关键词扩展）分别对接向量与 BM25，再 RRF 融合 + rerank——检索策略完整；
- **层次化摘要树**（L1~L4）混入主索引 + 查询期冗余过滤——解决"宏观问题查不到"的思路清晰；
- **知识图谱子系统**分层清楚（Extractor → Validator → Merger → Canonicalizer → Kuzu），且用不同模型交叉校验三元组，这是很成熟的做法；
- 双入口（CLI / Streamlit）通过 `rag/engine/query_engine.py` 共享查询配置，意图正确；
- config 中每个可选功能都有独立开关，调参友好。

### 1.2 模块健康度概览

| 模块 | 设计 | 健壮性 | 可维护性 | 主要问题 |
|---|---|---|---|---|
| `rag/retrieval/`（混合检索） | ★★★★ | ★★★ | ★★★ | 串行 LLM 调用、rerank 降级破坏分数 |
| `rag/indexing/`（分阶段索引） | ★★★★ | ★★ | ★★★ | 阶段完成检测不可靠、无原子写入 |
| `rag/summarization/`（摘要树） | ★★★★ | ★★ | ★★★ | 失败静默降级、并发异常中断全阶段 |
| `rag/graph/`（知识图谱） | ★★★★ | ★★ | ★★ | 校验逻辑失效、缓存自我失效、死代码多 |
| `rag/llm/`（LLM 工厂） | ★★★ | ★★ | ★★★★ | 无重试、伪流式 |
| `app/`（双入口） | ★★ | ★★ | ★★ | 50 行装配逻辑复制粘贴、ui 缓存 bug |
| `rag/config.py` | ★★ | — | ★★ | 446 行混杂密钥/端点/调参/prompt |
| 工程化（依赖/版本控制/测试） | ★ | — | — | 三者均缺失 |

## 二、高优先级缺陷（建议尽快修复）

以下是**正确性 bug**，按影响排序。每条附代码位置，可直接跳转。

### P0 —— 影响查询结果正确性或导致静默数据损坏

**1. 索引阶段"完成"检测只看目录是否存在 → 中断后静默加载损坏数据**
`rag/indexing/staged_indexer.py:362-367` 判断阶段是否完成的唯一依据是目录存在与否，但每个阶段都是**先建目录、后写文件**（如 `staged_indexer.py:103-105` 建 `chunks/` 目录在 `json.dump` 之前；`staged_indexer.py:271` 在向量阶段一开始就创建了 `data/vector/faiss`，等于提前创建了阶段标记）。中断后果：

- 摘要阶段写完 `summary_meta_map.json`、未写 `summary_nodes.json` 时中断 → `_load_summary`（`staged_indexer.py:188-203`）**对缺失文件静默返回空列表**，后续 BM25/向量索引里就永远没有摘要节点，无任何报错；
- 向量阶段 embedding 中途中断 → 下次启动认为向量阶段已完成，`_load_vector` 在缺 `default__vector_store.json` 时会**新建一个空 HNSW 索引**（`staged_indexer.py:315-320`）；
- 所有 `json.dump` / persist 均直接写最终路径，无"临时文件 + rename"的原子写入。

*建议：每阶段最后写一个 `_DONE` 标记文件（或 manifest 含节点数/时间戳），完成检测改为检查标记；所有 JSON 落盘改为 `tmp + os.replace`。*

**2. Streamlit 缓存与 log_list 交互 bug → 第二次查询起日志永远为空 + 内存泄漏**
`app/ui.py:44-45` 的 `load_index_and_engine(_log_list)` 被 `@st.cache_resource` 装饰，引擎持有的是**第一次调用传入的那个 list**。第二次查询传入新建的空 list（`app/ui.py:115-116`），缓存直接返回旧引擎：新 list 永远收不到日志（"运行流程"展开面板从第二次查询起全空），而被遗弃的第一个 list 持续累积日志、无人读取（内存泄漏）。`app/ui.py:114` 的注释"持有的是同一引用"与实际行为相反。

**3. 复杂度分类把"不是"判为"是" → 简单查询被误拆解**
`rag/retrieval/query_decomposer.py:96`：`return "是" in response or ...`。LLM 回答"不是"（中文最自然的否定）时包含"是"字 → 误判为复杂查询，触发一次不必要的拆解 LLM 调用 + 多路子查询全管线检索，**静默把查询延迟放大数倍**。应改为前缀匹配或让 LLM 输出结构化标记。

**4. reranker 降级把全部分数置 0 → 子查询合并排序失效**
`rag/retrieval/reranker.py:67 / 74 / 91` 三处降级路径返回 `(i, 0.0)`，`hybrid_retriever.py` 的 `_do_rerank`（约 319-323 行）随即把所有节点 `score` 覆写为 0。下游 `_decomposed_retrieve`（约 211-215 行）按 score 排序合并、按 score 去重——全 0 之后排序等于随机、去重保留任意一条。降级应保留 RRF 分数而非清零。

**5. 预处理丢正文：章节标题与第一个小节标记之间的文本被丢弃**
`rag/ingestion/preprocessor.py:153-165`：`_split_body_by_subsections` 只收集每个小节标记 `sm.end()` 之后的文本，章首、第一个小节标记之前的正文**不进入语料**——这部分内容既不可检索、也不进摘要和图谱。

**6. 图谱校验体系大面积失效**
- `rag/config.py:357` `GRAPH_VALIDATE_CONFIDENCE_THRESHOLD = 2.0`（注释说"设为 2.0 表示全部校验"），置信度上限是 1.0，因此**所有关系都是"低置信度"**，`extractor.py:227-247` 整套置信度打分成为死代码；
- 更严重的是 `rag/graph/validator.py:79-84`：低置信度关系只有 1 条时**直接放行、跳过 LLM 校验**——单关系 chunk 永远不会被校验；
- `validator.py:121` 假定 LLM 返回的 `valid` 下标是 int，LLM 返回 `["0","2"]`（字符串）时全部匹配失败 → 该 chunk 的低置信度关系**全部被丢弃**（而 `reasons` 却按 `str(i)` 取，`validator.py:133`，自身就不一致）。

**7. 图谱缓存"自我失效"：schema 自动成长导致全量重抽**
Schema fingerprint 把 `learned_types` 算进去（`rag/graph/schema.py:219`），而 learned types 在构建过程中会增长并保存（`graph_constructor.py:254`）。下次运行加载成长后的 schema → fingerprint 变化 → `graph_constructor.py:112-118` **删除整个 SQLite 缓存、全书重新抽取**。断点续传机制被自己的 schema 成长机制击穿。

**8. 抽取主循环无逐 chunk 兜底 → 一个 chunk 崩掉整个图构建**
`rag/graph/extractor.py:91-96`：`_try_json_repair` 可能返回 list/str（LLM 输出 JSON 数组时），`if not data` 对非空 list 不生效，接着 `data.get(...)` 抛 `AttributeError`；而主循环 `graph_constructor.py:165-251` **没有 per-chunk try/except**，异常直接终止整个构建，该 chunk 卡在 `processing` 状态。另外 `extractor.py:106-107`：只有实体、没有关系的 chunk 返回 `None`，被标记为"失败"，实体描述被丢弃且每次续跑都重抽。

**9. 图检索 Cypher 注入/中断**
`rag/graph/graph_retriever.py:127-133` 用 f-string 把 LLM 提取的实体名直接拼进 Kuzu 查询的单引号里——实体名含 `'` 即查询报错（被静默 `continue` 吞掉）。应使用参数化查询。

**10. canonicalizer 记账错误 → 中途新增实体不参与归一化**
`graph_constructor.py:216-217`：守卫条件判断的是 `orig_name not in all_known_entities`，append 的却是 `canonical`（本来就在列表里 → 重复），而真正的新实体（`canonicalize` 返回 `None` 的）**从未被 append**——构建中途出现的新实体对后续 chunk 的归一化不可见。另外 `graph_constructor.py:210` 在 `save_entities`（`:242`）之前调用 `update_entity_canonical`，当前 chunk 的行还没入库，canonical 映射对它们丢失。

### P1 —— 影响稳定性 / 结果质量

**11. DavyLLM 无重试、无 429 处理**
`rag/llm/factory.py:97-105` 单次 `requests.post` + `raise_for_status()`。config 自己注明"并发 >2 会触发 429"（`config.py:219`），但全代码库没有任何重试/退避/`Retry-After` 处理；抽取、校验、摘要、改写全部是单发即弃。摘要树/图构建这类数千次调用的长任务，一次瞬时 429 就丢一个 chunk 的结果。响应解析 `resp["choices"][0]["message"]["content"]`（`factory.py:111`）遇到异常 payload 直接 KeyError，`content` 为 None 时 `_strip_thinking(None)` 抛 TypeError（`factory.py:74`）。

**12. 摘要失败时用原文截断"冒充"摘要，且无任何标记**
`rag/summarization/summary_tree.py:124`（叶子：`node.text[:100]+"..."`）、`:339-342`（父级：`source_text[:200]+"..."`）。LLM 调用失败后，截断原文被当作摘要**进入索引**，无 metadata 标记、无统计计数——检索质量被静默污染，事后无法定位哪些摘要是假的。另外 `summary_tree.py:202 / 401 / 472` 的 `future.result()` 未捕获异常：一个 worker 崩溃将中止整个摘要阶段，已完成的全部 LLM 调用成果丢失（阶段内无增量持久化）。

**13. BM25 阶段覆写摘要节点的 `original_text` 元数据**
`staged_indexer.py:224-227` 把所有节点的 `original_text` 覆写为 `section + 节点文本`，而摘要树在 `summary_tree.py:660` 特意把**原始出处文本**放进 `original_text` 供 rerank/展示使用——被覆盖后 reranker 对摘要节点评的是摘要本身而非原文。

**14. 查询引擎给图谱上下文固定 score=1.0**
`rag/engine/query_engine.py:38-44`：注入的图谱节点 score=1.0，高于几乎所有 rerank 分数，任何按分数排序/截断的下游逻辑都会永远优先图谱节点。

**15. `stream_chat` 是伪流式**
`factory.py:118-159`：把整个 SSE 流读完、`_strip_thinking` 后一次性 yield 一个 chunk——UI 端的"流式输出"实际是全量阻塞等待。`_strip_thinking`（`factory.py:74`）只处理 `<thinking>` 标签，Qwen/DeepSeek 惯用的 `<think>` 不会被剥离。

**16. 其他值得注意的正确性问题（简列）**

| 位置 | 问题 |
|---|---|
| `preprocessor.py:46-56` | L1 标题格式只采样前 3000 字符判断；长序言可导致全书判型错误 |
| `preprocessor.py:27` | `_SUB_PATTERN` 实际也匹配"第1节"，与自身注释矛盾 |
| `summary_tree.py:533-535` | `_section_order` 用 `\d+` 提取序号，"第十二回"这类中文数字全部得 0 + `as_completed` 乱序 → L4 全书摘要的输入顺序不确定，**两次构建产物不一致** |

| `cli.py:110` / `ui.py:125` | `query_engine.query()` 无任何异常处理，一次网络失败即崩掉整个（耗时数分钟加载的）会话 |

## 三、安全与配置问题

1. **API key 明文硬编码**：`rag/config.py:38`（`DAVY_API_KEY = "..."`)。建议改为 `os.environ.get("DAVY_API_KEY")` + `.env`（配合 `.gitignore`），部门内共享代码前必须处理。
2. **内网基础设施全部硬编码**：`config.py:11`（embedding 端点）、`:34`（Davy URL）、`:197`(reranker)、`:349`（校验 LLM）。换机器/换环境需要改源码。
3. **TLS 证书随代码走相对路径引用**（`config.py:37` + `assets/`），文件缺失时 requests 报出的 OSError 极难排查，建议启动时做一次存在性检查并给出明确报错。
4. **`DEBUG = True` 是提交的默认值**（`config.py:446`）——每次正式查询都输出 top-k 明细。
5. **config.py 应分层**：目前 446 行混杂"基础设施端点 + 密钥 + 调参 + 约 300 行 prompt 模板"。建议拆为：
   - `config.py`——纯参数（可被环境变量覆盖）；
   - `prompts.py`（或 `prompts/*.txt`）——全部 prompt 模板；
   - `.env` / 环境变量——密钥与端点。
   现在的 `.format(novel_context=...)` + `{{query}}` 双层转义写法（`config.py:76-130`）也容易出错，拆出去后可消除。

## 四、可维护性 / 重构建议

按"现状 → 建议"列出，均为**局部重构**，不动整体架构。

### 4.1 消除双入口复制粘贴（收益最大）

`app/cli.py:57-105` 与 `app/ui.py:52-96` 的整段引擎装配逻辑（约 50 行：top-k 兜底、Reranker/QueryRewriter/QueryDecomposer/HybridRetriever/GraphRetriever/query_engine 组装）逐行相同；引用渲染循环（`cli.py:124-143` vs `ui.py:134-154`）、Settings 初始化（`cli.py:45-50` vs `ui.py:34-40`）同样重复。任何管线改动都要改两处，且已经出现漂移（ui 的缓存 bug 即源于此）。

**建议**：新建 `rag/engine/bootstrap.py`，提供 `build_query_engine(log_list) -> (engine, ...)` 与 `format_source_nodes(nodes) -> list[str]` 两个函数，两个入口各留 20 行以内的壳。

### 4.2 log_list 模式 → 标准 logging

`_log` 闭包/方法在全代码库复制了 **20 处左右**（`staged_indexer.py` 9 处、`summary_tree.py` 5 处、各检索类 5 处、`graph_constructor.py` 2 处……），本质是把一个可变 list 穿透 6+ 层构造函数当穷人版 logger，且与真正的 `logging` 并存双轨、非线程安全（Streamlit 会话共享缓存引擎时是真实风险）。

**建议**：定义一个自定义 logging Handler（把记录收集到当前请求上下文，如 `contextvars`），所有模块回归 `logger.info(...)`；入口层从 handler 取日志渲染。一次性消灭全部 `_log` 样板 + 解决线程安全 + 顺带修复 ui 缓存 bug。

### 4.3 抽公共工具

- **JSON "repair-then-regex" 解析写了 4 遍**：`extractor.py:34-51`、`validator.py:20-36`、`canonicalizer.py:99-110`、`graph_retriever.py:36-52` → 抽成 `rag/utils/json_parse.py`，并顺带修复"返回值可能是 list/str"的类型问题（缺陷 #8）。
- **LLM 调用 + 重试**：给 `DavyLLM._call` 增加指数退避重试（尊重 `Retry-After`），全部调用方自动受益（缺陷 #11）。
- `tokenize_for_bm25` 定义了两份（`indexer.py:21` 是死代码，注释引用的 `retriever.py` 已不存在）；序列化助手两组四个函数形状相同（`staged_indexer.py:43-90`）可合并。

### 4.4 删除死代码与死配置

| 类别 | 明细 |
|---|---|
| 兼容壳 | `indexer.py` 整个文件只为转发 `get_or_build_index`，入口可直接 import `rag.indexing.staged_indexer` |
| 死配置 | `RERANK_MAX_CONCURRENCY`（注释里的线程池不存在）、`GRAPH_RETRIEVAL_DEPTH`（检索只有 1 跳）、`GRAPH_CANONICALIZE_ENABLED`（从未被检查，归一化无条件执行）、`GRAPH_CHUNK_SIZE/OVERLAP`（已被按节抽取取代）、`SUMMARY_GROUP_SIZE`、`SUMMARY_LEAF_PROMPT`、`GRAPH_SCHEMA_GROWTH_THRESHOLD`（定义了但从未传给 `Schema()`）、`DAVY_MODEL_NAME_GPT_OSS` |
| 死代码 | `preprocessor.py:59-89`（`_detect_sub_section_level`/`_build_section_path`/L2/L3 正则，section_path 恒为标题）、`preprocessor.py:305,316-329`（`forced` 永远不为 True）、`cache.py` 约 8 个从未调用的方法（`:156/:182/:221/:257/:313/:322/:354-374`）、`graph_constructor.py:291-295`（`relation_index` 建了不读）、`summary_tree.py:55-57`、`query_decomposer.py:132`（恒等表达式） |

| 过时注释 | `query_engine.py:6`、`ui.py:119` 仍引用早已改名的 `app.py`/`app_ui.py` |

### 4.5 语料无关化（如果计划复用到其他文档）

当前"通用"模块里烧死了《遥远的救世主》专有内容：`rag/graph/rules.json:24-27` 硬编码人物名单（`known_male_characters`），config 中的改写/摘要/抽取 prompt 全部点名小说与人物。若这个系统未来要给部门其他文档用，需要把"语料档案"（背景介绍、术语表、人物规则、示例）抽成独立的数据文件，prompt 模板只留占位符。若明确只服务这一本书，此项可不做。

## 五、性能优化机会（按收益排序）

1. **三路改写串行**：`query_rewriter.py:135-137` NL/HyDE/关键词三次 LLM 调用严格串行（模块 docstring 却写着"三路并行"）——三路彼此独立，`ThreadPoolExecutor` 并行即可把改写延迟降为 1/3。这是**单次查询延迟的最大头**。
2. **子查询串行全管线**：`hybrid_retriever.py:205-207` 拆解后的每个子查询串行跑完整管线，最坏 5 个子查询 ≈ 15 次串行 LLM 调用 + 15 次检索 + 5 次 rerank。子查询之间可并行。
3. **每个简单查询也多付 1~2 次 LLM 往返**：启发式判不出来就调 `_llm_classify`（`query_decomposer.py:72-76`），图检索的实体提取（`graph_retriever.py:106`）又是一次阻塞调用且排在主检索之后——后者可与主检索并发。
4. **CLI 每次查询全量重建**：`cli.py:36-105` 每次 `run_query` 都重新加载 FAISS/BM25/摘要/图谱。若 CLI 有交互式/批量使用场景，应加载一次、循环提问（顺带把 `cli.py:148` 硬编码的问题改成 `sys.argv`/stdin）。
5. **图构建串行且 merge 调用放大**：整个图构建一节一节串行；`graph_constructor.py:220-233` 对每条关系的每个端点都可能触发一次 merge LLM 调用——同一实体出现在 5 条关系里 = 单 chunk 最多 10 次 merge 调用。可按实体去重后再 merge，并小并发（注意 429，配重试后再开）。
6. **embedding 无 checkpoint**：向量阶段 5000 个 chunk 崩在第 4999 个 = 全部重来（`embedding_progress.py` 只做进度打印不做续传）；`EMBED_BATCH_SIZE=512` × 4096 维意味着单个 Ollama 请求体非常大，建议降批次 + 分段持久化。
7. **BM25 阶段 3 倍内存**：`staged_indexer.py:219-228` 深拷贝所有节点又把全文再塞进 `original_text` metadata；全书 + 摘要全程驻留内存。当前单本小说规模无所谓，扩大语料前需要改。
8. `GraphCache` 几乎每条语句一次 `commit`（`cache.py:131` 起多处）——热循环里应批量提交。

## 六、工程化缺失（建议最先补齐）

这部分不改一行业务代码，但对"部门内部使用"的定位最重要：

1. **没有版本控制**：目录不是 git 仓库。`git init` + 合理的 `.gitignore`（`data/`、`__pycache__/`、`.env`）应该是所有改动之前的第一步——否则后续任何重构都没有回退手段。
2. **没有依赖清单**：`requirements.txt`/`pyproject.toml` 均缺失，依赖（`llama-index` 系列、`faiss`、`kuzu`、`jieba`、`streamlit`、`docx2txt`、`json_repair`、`networkx`、`pyvis` 等）全靠环境记忆——本机系统 Python 目前甚至装不齐这些包，说明环境已经漂移。用 `pip freeze` 固定一份是零成本的。
3. **没有任何测试**：仅有遗留的 `rag/graph/_test_graph_db/`。不必追求覆盖率，但至少给三类纯函数补最小单测（不需要网络/LLM）：
   - `preprocessor` 的章节切分（直接能锁定缺陷 #5）；
   - `_parse_batch_leaf_response`、JSON 解析工具、validator 的下标解析（锁定缺陷 #6/#8）；
   - RRF 融合与 gap 过滤。
   当前模块对全局 `config`/`Settings` 的耦合是可测性的最大障碍，4.1/4.2 的重构会同时改善这一点。
4. **入口体验**：`app/cli.py` 问题硬编码在源码尾部（`cli.py:147-148`），应支持 `python -m app.cli "问题"`。
