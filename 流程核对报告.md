# 流程核对报告 ——《流程.txt》vs 实际实现

> 核对日期：2026-07-23，基于当前 main 分支代码（188 例离线单测全过）。
> 逐条对照《流程.txt》与实际代码，标注：✅ 一致 / 🟡 部分一致（实现方式或细节不同）/ ❌ 不一致（文档描述的能力未实现或实现相反）。
> 文档自称"大致流程"，故实际代码**多做**的优化不算不一致，在文末单独汇总。

## 总体结论

主体流程与文档一致：章节感知分块 → 摘要树/BM25/向量/图谱四类索引 → 查询分解 + 三路改写 + 混合检索 + 图检索 → 生成答案 + 参考文献，全部落地。

实质性差异共 **3 处**（1.3 逗号粒度、2.2.3 L3 生成来源、3.1/4.1 子查询结果送 LLM 的方式），另有若干实现方式不同但达成同等目的的条目，以及 2 处文档编号笔误。

## 一、分块（流程 1）

| 条目 | 结论 | 说明 |
|---|---|---|
| 1.1 加载文档 | ✅ | `rag/ingestion/preprocessor.py:load_documents`，支持 `.txt` / `.docx`（`data/raw/`） |
| 1.2 章节感知到最小层级 | ✅ | 自动检测一级标题风格（「一、」vs「第X回/章/节/篇」），再按小节标记（「　　1」/「第X节」）二级拆分；章节标题与首个小节标记之间的正文保留为 `subsection=""` 块 |
| 1.3 逐级分块（章→节→句→**逗号**） | 🟡 | 实际是**滑动窗口 + 语义边界右扫描**：切点必须落在句末标点（。！？……及 。"/！"/？"/"。/）。/……" 组合）之后。**没有逗号级兜底**——一句超长无句末标点的文本会在 `chunk_size` 处硬切，最小粒度是"句"而非文档所述"逗号" |
| 注① 新字数约 1000、先规划块数再微调防小块 | 🟡 | `CHUNK_SIZE=1024` ≈ 1000 ✅；防小块的手段不是"全局预规划块数"，而是**尾块均衡**：剩余文本 ≤ 2×chunk_size 时拆成两个大小接近的块（`_do_tail_balance`），达到同等目的 |
| 注② 语义边界、引号内不分块、不跨章节 | 🟡 | 语义边界 ✅；不跨章节 ✅（每个章节/小节独立分块）。"两个引号之间不分块"是**近似实现**：优先匹配双/三标点组合避免把 。"等切开，但引号**内部**的 。+汉字 仍可能成为切点，不保证引号内绝不分块 |
| 注③ 滑动窗口，单侧重叠 100–150 字（10%–15%） | 🟡 | `CHUNK_OVERLAP=102`（单侧**上限**约 10%，处于文档区间下限）；且重叠是动态的——左边界从 `prev_R−102` 向右扫描语义边界，实际重叠 ≤ 102 字。与注②冲突时优先语义边界 ✅ |

## 二、构建索引（流程 2）

| 条目 | 结论 | 说明 |
|---|---|---|
| 2.1→2.4 阶段顺序 | 🟡 | 文档顺序：BM25 → 摘要树 → 向量 → 图谱；实际顺序（`staged_indexer.py`）：**分块 → 摘要树 → BM25 → 向量 → 图谱**。BM25 必须晚于摘要树，因为**摘要节点也混入 BM25 索引**（文档 2.3.1 只提向量索引包含摘要）——这是实际比文档多做的一点 |
| 2.2.1 L1 叶子摘要（~100 字） | ✅ | 每 chunk ≤100 字一句话摘要；实现上每 5 个 chunk 合并为一次 LLM 调用（`SUMMARY_LEAF_BATCH_SIZE=5`），批间并发 2 |
| 2.2.2 L2 小节摘要（原文×20%，与叶子无关） | ✅ | `use_original=True` 直接读小节原文，`SUMMARY_PARENT_RATIO=0.20`，与 L1 完全独立 |
| 2.2.3 L3 章节摘要 | ❌（来源）/ 🟡（单小节晋升） | 文档："根据该章节的**所有 L2 小节摘要**构建 L3"；实际：`_generate_chapter_summaries` 以 `use_original=True` 从**章节原文**（各 L2 的 `original_text` 拼接）生成，15%（`SUMMARY_CHAPTER_RATIO`）作用于原文而非摘要。"只有 1 个小节时 L2 直接当 L3"：实际条件是"该章**无小节标记**（`subsection==""` 且仅 1 个 L2）"——若某章恰有 1 个带编号小节，仍会生成 L3，与文档略有出入 |
| 2.2.4 L4 全书摘要（~500 字） | ✅ | 由全部"章节级"节点（L3 + 无小节章的 L2）的**摘要文本**合并生成（`use_original=False`），`SUMMARY_BOOK_CHARS=500`；按章节序号显式排序保证两次构建产物一致 |
| 2.3 为所有块和所有摘要建向量 | ✅ | chunk + 全部摘要节点混入向量索引（FAISS HNSW，qwen3-embedding:8b，4096 维，embedding 分段断点续传——文档未提） |
| 2.4.1 图谱按小节分块 | ✅ | 图抽取直接用 `load_documents()` 的章节/小节文档，不单独分块 |
| 2.4.2 抽取实体/关系 | ✅ | `rag/graph/extractor.py`，召回优先 |
| 2.4.3 规则过滤 | ✅ | `rag/graph/rules.json` |
| 2.4.4 LLM 二次校验（暂全量） | ✅ | `GRAPH_VALIDATE_CONFIDENCE_THRESHOLD=2.0` = 全部关系送校验；校验用**与抽取不同的模型**交叉检查（gpt-oss-120b，文档未提），可纠正三元组 |
| 2.5.5 增量合并、全局去重 | ✅ | `DescriptionMerger`（描述增量合并）+ `Canonicalizer`（别名→规范名，确定性快速规则优先于 LLM）；chunk 内按唯一实体去重，每实体每 chunk 最多 merge/canonicalize 一次。（编号笔误：2.5.5 应为 2.4.5） |
| 注① 所有摘要记录其对应输入 | ✅ | 各级 `original_text`：L1/L2 记原文，L4 记输入的编号子摘要；**L3 记录的是章节原文**——与其实际生成来源一致，但与文档"L3 记录其输入的摘要"的表述对应不上（根因即 2.2.3 的差异） |
| 注② 抽取断点续传 | ✅ | SQLite per-chunk 缓存（`data/graph_cache/`），指纹含实际模型名；单 chunk 失败不中止全局；`--force` 全量重建。另外 embedding 也做了断点续传（文档未提） |

## 三、多路检索（流程 3）

| 条目 | 结论 | 说明 |
|---|---|---|
| 3.1 查询分解、子查询并行走完整流程 | 🟡 | 复杂度判断（启发式 + LLM 前缀匹配兜底）→ LLM 拆分（≤5 个）→ 子查询**并行**各走完整管线 ✅。但"将所有子查询的**结果**统一送入 LLM"的方式不同：实际是把各子查询的检索结果按**子查询内名次做 RRF 融合**（对分数量纲免疫），取 top-10 片段后与**原始查询**做一次答案合成——**不会**把"每个子查询 + 其结果"逐个标注送入 LLM（对应流程 4.1 括号内的描述同样未按字面实现） |
| 3.2.1 术语映射 | ✅ | `assets/terminology.json`，最长匹配优先、单遍正则替换（避免级联替换）；改写关闭时也生效 |
| 3.2.2 三路改写并行 | ✅ | NL 改写 / HyDE / 关键词扩展并行（并发 2），全部以小说背景块 `NOVEL_CONTEXT` 接地；单路失败回退映射后查询 |
| 3.2.2.1.1 分路检索 | ✅ | NL、HyDE → 向量（top-30），关键词 → BM25（top-30）；BM25 查询串统一先过 jieba 分词（文档未提，关键细节——语料是分词后的，整句中文查询得分为 0） |
| 3.2.2.1.2 摘要冗余过滤（三路整体） | ✅ | 三路 node_id 取**并集**判断，命中摘要覆盖的 chunk ≥50% 即删摘要，再对每路各自过滤 |
| 3.2.2.1.3 每路相邻分数比过滤 | ✅ | gap>0.2 截断，每路 10–30 条；另加每路 min-score 下限（`VECTOR_MIN_SCORE`/`BM25_MIN_SCORE`，文档未提） |
| 3.2.2.1.4 三路 RRF 融合 | ✅ | k=60 |
| 3.2.2.1.5 三路整体重排序 | ✅ | RRF top-20 送 bge-reranker-v2-m3，取 top-10；瞬时故障快速重试 1 次，仍失败降级为 RRF 顺序（文档未提降级路径） |
| 3.2.2.2 图检索与主检索并行 | ✅ | `GraphAugmentedQueryEngine` 中与整条三路管线并发执行 |
| 3.2.2.2.1 提取实体或关系 | ✅ | LLM 从查询提取具名实体（仅实体，不提关系——够用） |
| 3.2.2.2.2 图数据库查询 | ✅ | Kuzu 参数化 `CONTAINS` 匹配（每实体一跳三元组，全局去重，上限 20 条）；结果以"当前最低分"作为附加上下文注入，不压过 rerank 结果 |

## 六、实际实现比文档多做的部分（文档未覆盖的优化）

- **工程化**：五阶段独立持久化 + `_DONE.json` 完成标记 + 原子写入，删目录即可重建单阶段；embedding 分段断点续传（`data/embed_cache/`）。
- **索引**：摘要节点同时混入 BM25 与向量索引；元数据排除键防止 `original_text`/UUID 列表污染嵌入输入与 QA prompt；FAISS HNSW 加速检索。
- **图谱**：有界流水线并发（worker 抽取+校验，主线程串行记账）；校验用不同模型交叉；Schema 自动成长（learned 类型不进指纹，不破坏续传缓存）。
- **检索健壮性**：BM25 查询侧统一 jieba 分词兜底；rerank 失败降级 RRF；子查询名次 RRF 融合免疫分数量纲差异；每路 min-score 下限。
- **LLM 层**：Davy 429/5xx 指数退避重试（尊重 Retry-After）；`<think>` 块增量剥离的真流式；断流未出正文时整流重试。
- **可观测性**：contextvars 日志捕获（Streamlit 下每查询独立日志）、`RAG_DEBUG` 分步 top-k 详情、降级摘要显式标记与统计。

## 七、建议

1. 若希望文档与实现完全对齐，可修订《流程.txt》的三处实质差异（1.3 逗号粒度、2.2.3 L3 来源、3.1 子查询结果合成方式）与两处编号笔误；
2. 若"逗号级最小粒度"和"引号内绝不分块"是硬需求，需在 `_find_semantic_boundary_right` 中增加逗号兜底与引号配对状态跟踪——当前实现未做。
